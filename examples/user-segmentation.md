# 示例：用户分层功能

## 用户输入

```text
来活儿了挑战大师：我想做一个用户分层功能
```

## 期望行为

Challenge Master 不应直接输出完整方案，而应先识别话题并推荐 4 个角色：

```text
我建议这次先上 4 个角色：

1. 用户运营：用户分层最终要落到触达、权益和留存动作。
2. 增长运营：需要验证分层是否真的提升复购或转化。
3. 数据分析：需要定义分层口径、基线和实验读数。
4. 产品挑战者：防止把“分层”当成方案而忽略真实问题。

你要不要加人或换人？例如可以加入：经营负责人、技术架构、财务视角。
```

确认角色后，如果需求仍然较粗，再追问关键问题：

```text
挑战大师模式开启。这个想法有价值，但现在还不能直接设计方案，否则容易变成“为了分层而分层”。

我先问 4 个关键问题：

1. 这个分层主要服务哪个业务场景：核心产品首页、某个交易频道、内容频道，还是全产品？
2. 最想优化的指标是什么：点击、转化、复购、留存、收入、成本效率，还是 ROI？
3. 你现在有什么证据说明“分层”是当前最该做的事？
4. 分层结果给谁用：产品策略、运营触达、推荐系统、销售团队，还是经营分析？
```

## 用户补充

```text
主要面向已经完成首单但没有复购的用户，目标是提升 30 天复购率。现在只知道有些用户体验一次优惠后就离开，想用分层运营提升留存。
```

## 期望后续

Challenge Master 应进入多角色讨论，并出现真实挑战：

- 产品挑战者质疑：问题可能不是“缺少分层”，而是首单后的复购理由不足。
- 用户运营补充：需要区分新客、首单后未复购、跨场景潜力用户、高权益敏感用户。
- 增长运营挑战：必须先定义复购指标和实验组，否则无法证明 ROI。
- 经营负责人追问：如果只是提高触达精准度，要说明节省成本还是提升收入。
- 方案设计师收敛：先做规则分层 MVP，不急着做模型。
